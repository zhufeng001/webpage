postgresql9.1准实时同步测试  

2011-12-08 18:07:21|  分类： postgresql |  标签： |举报 |字号大中小 订阅

 

Postgresql9.1实时主从测试

Master服务器配置

CPU:2U4核

内存:6GB

 

Slave 服务器配置

CPU:2U4核

内存:2GB

 

网络连接为100M/s局域网

 

1.      Master数据库启动，Slave数据库关闭

场景当master 启动slave关闭时在master端执行create table test (id serial8,name varhcar(30));

现象：master客户端执行过SQL后一直处于事务未结束状态。

 

Ctrl+c结束当前事务,执行select * from test;

现象：master端test表已建立

在Master端执行insert into test (name) values(‘che’);,Crtl+c并执行\q退出客户端。启动Slave数据库执行select * from test;

现象:test表已存在且在Master中插入的数据Slave中也存在。

 

结论:如在Slave端不存在则Master 端数据库连接将出现异常，但操作输入依然能插入Master数据库，待Slave连接正常后数据会同步到Slave端。如Xlog文件被覆盖则Slave端将无法保证数据同步。

 

2.      Master数据库正常启动、Slave数据库关闭，对Master数据库向test表中插入三条数据，插入后关闭Master数据库，而后将Slave和Master均启动。在Master和Slave中执行Select * from test。

现象:Master和Slave表中数据一致，刚插入的记录在两个库中均存在。

结论：如Slave连接异常，Master有数据进库后出现异常数据库关闭则两个数据库重启正常后数据仍然一致。

 

3.      Master与Slave数据库均启动且连接正常，在Master客户端调用函数向test表中插入200万条数据，插入完上后马上关闭Master数据库服务器网卡。

现象：查看Master与Slave端数据库中数据均为200万条。

结论：9.1的主从能保证事务的一直性

 

4.      Mater与Slave数据库均启动，单Mater与Slave连接断开，在Master端执行truncate test；后Ctrl+c强制结束。

现象:Master数据库中数据被清空，Slave中数据任然存在。

Mater与Slave连接正常后，立即在Slave端执行select * from test;仍然能查到数据，再次执行select * from test则数据表为空。

结论：9.1的主从为在Master与Slave数据库服务器端事务的基础上又包了一层，并不是真正的一个事务，但是如有异常恢复后Mater与Slave根据xlog可以自动同步。

 

5.      Master数据库服务器关闭，Slave数据库服务器运行正常。登录Slave数据库执行insert into test (name) values(‘test’);

现象：提示ERROR:  cannot execute INSERT in a read-only transaction

将Slave数据库服务器postgresql.conf文件中hot_standby参数值修改为off，并将recover.conf文件删除，重启Slave数据库后可对Slave数据库执行insert into test (name) values(‘test’);

结论：如Master数据库服务器出现问题则将Slave服务器hot_standby参数值修改为off，并将recover.conf文件删除即可为应用程序提供服务。

 

6.      将经过第5个测试场景的主从数据库还原为可正常进行Master Slave实时同步的配置。

现象：Master数据库可正常启动，Slave数据库启动时报以下错误。

LOG:  database system was shut down at 2011-12-08 17:04:11 CST

LOG:  entering standby mode

WARNING:  WAL was generated with wal_level=minimal, data may be missing

HINT:  This happens if you temporarily set wal_level=minimal without taking a new base backup.

FATAL:  hot standby is not possible because wal_level was not set to "hot_standby" on the master server

HINT:  Either set wal_level to "hot_standby" on the master, or turn off hot_standby here.

LOG:  startup process (PID 11760) exited with exit code 1

LOG:  aborting startup due to startup process failure

结论：Slave服务器变为可读写状态进行修改操作后，无法还原原有的Master―Slave架构。

7.      将Master数据库服务器postgresql.conf中synchronous_commit分别设置为ON和Off

ON状态插入100万条记录系统耗时如下

结论：如果Master与Slave服务器网络连接无问题则，则准实时同步和实时同步性能无差别。

8.      Master-Slave实时同步正常运行，在Slave端获取一个Sequence的下一值select  nextval('test_id_seq');

现象：ERROR:  cannot execute nextval() in a read-only transaction

结论：9.1处于Master-Slave实时同步时Slave数据库处于自读状态


9.      在Master数据服务器中初始化一个5433端口的数据库系统，5433不设置主从复制，通过pgbench测试两个数据库系统10个客户端连接是处理能力
